# Схема защиты и управления доступом

Механизм аутентификации JWT (JSON Web Tokens) в связке с OAuth 2.0 (для MVP - развернутый Keycloak,
в дальнейшем интегрироваться с такими системами как Яндекс, VK)

## Архитектура аутентификации

[Клиенты: Web/Mobile]
↓ (OAuth2)
[Auth Service]
↓ (проверка, создание сессии)
[Выдача JWT]
↓ (хранится у клиента)
[API / Микросервисы]
↓ (проверка подписи JWT)
[Доступ разрешён]

Существует два вида токенов: токен доступа (access token) выдается на 1 час, токен обновления (refresh token) - 1 месяц.
По истечении времени жизни токена доступа производится обновление токенов с использованием токена обновления.
Токен доступа передается в заголовке с ключом Authorization и значением Bearer {access_token}.
AuthService (Keycloak) при авторизации формирует токен доступа и токен обновления, отправляет токен доступа
пользователю, токены асинхронно кладутся в Redis (для быстрой проверки) и БД (для долгосрочного хранения).

## Безопасность передачи данных

Цель безопасности:

1. Шифрование трафика (между клиентом и сервером, между микросервисами)
2. Аутентификация сервера (и опционально клиента)
3. Целостность данных
4. Защита от MITM, downgrade-атак, утечек сессий

Следующие данных обязательны для шифрования (даже при TLS)

| Категория данных                   | Метод                      | Алгоритм    |
|------------------------------------|----------------------------|-------------|
| Персональные данные (ФИО, телефон) | Шифрование                 | AES-256-GCM |
| Финансовая информация              | Токенизация или шифрование | AES-256-GCM |
| Refresh-токены                     | Случайный токен            | -           |
| Секреты приложения                 | Шифрование                 | AES-256-GCM |
| Чаты                               | Шифрование                 | AES-256-GCM |

## Управление доступами

Реализуем модель управления доступом на основе RBAC (Role-Based Access Control) с возможностью расширения
до ABAC (Attribute-Based Access Control) в будущем.

| Роль    | Описание             | Разрешения                                                                      |
|---------|----------------------|---------------------------------------------------------------------------------|
| user    | Обычный пользователь | ad:create, ad:read, ad:update:own, ad:delete:own, favorite:manage, order:create |
| admin   | Администратор        | Все права + ad:moderate, user:block, review:moderate                            |
| service | Системный сервис     | ad:read, user:read, event:publish — только по API, без сессии                   |

Разрешения (информация о них включается в JWT)
ad:create       Создание объявления
ad:read         Просмотр любого объявления
ad:update:own   Редактирование своего объявления
ad:update:any   Редактирование любого (для admin)
ad:delete:own   Удаление своего
ad:delete:any   Удаление любого
ad:moderate     Модерация (скрыть, заблокировать)
user:read       Просмотр профиля пользователя
user:block      Блокировка пользователя
review:moderate Удаление/одобрение отзыва
favorite:manage Работа с избранным
order:create    Создание заказа
event:publish   Публикация событий в шину (для сервисов)

## Управление секретами

Для безопасного управления секретами в распределённой микросервисной системе (K8S) лучшим решением является HashiCorp
Vault.
