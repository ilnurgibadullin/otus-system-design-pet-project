# Форматы взаимодействия

Взаимодействие разбито на три слоя: client layer (пользовательский слой), edge layer ("пограничный" уровень) и services
(сервисный слой).

Client Layer <-> Edge Layer:
Для MVP будет реализовано только APP клиент через который пользователи смогут взаимодействовать с приложением. 
Основной протокол HTTP/1.x, после успешного запуска рассмотреть переход на целевое взаимодействие HTTP/2.
Функция чата будет реализована посредством WebSocket-взаимодействия, так как общение между продавцов и покупателем
должно быть оперативным.
Push-уведомления получаем по HTTP/1.x, интегрируясь с FCM (для Androind) или APNs (для iOS).

Edge Layer <-> Client Layer:
Основным протоколом выступает GRPC. Формат взаимодействия между API Gateway и нашими сервисами будет стандартизирован,
позволяет легко версионировать будущие изменения в контрактах. Более того, объем одного сообщения по сравнению с http ниже,
тем самым снижаем объем передаваемого трафика.

Пример RestApi в формате в директории openapi.

## Service Discovery:
Когда микросервисы развернуты в динамической среде (например, в Kubernetes или Docker), их IP и порты могут меняться.
Service Discovery решает эту проблему:

Регистрация сервиса: при запуске микросервис регистрируется в центральном реестре сервисов (service registry),
указывая своё имя, IP, порт и статус.
Обнаружение сервиса: другой микросервис запрашивает у реестра актуальный адрес нужного сервиса по имени.
Балансировка нагрузки: клиент может выбирать один из нескольких экземпляров сервиса.
Здоровье сервиса: реестр проверяет "живость" сервисов (через health checks) и удаляет неработающие.

## API Gateway
Настройка API Gateway — ключевой элемент архитектуры микросервисов. Он выступает как единая точка входа, обеспечивая:
Маршрутизацию запросов к нужным микросервисам
Аутентификацию и авторизацию (CORS)
Ограничение скорости (Rate Limiting, Circuit Breaker)
Логирование, мониторинг, кэширование

## Сценарии сбоев и стратегии обработки

### Отказ одного из микросервисов (например, advertisement-service)

Проблема:
Один экземпляр (pod) advertisement-service падает.
Другие экземпляры продолжают работать.
Решение:
Service Discovery удаляет неработающий экземпляр из реестра после провала health check.
API Gateway или клиент получает только список живых экземпляров.
Балансировщик нагрузки (например, Nginx) перенаправляет трафик на оставшиеся инстансы.
Повторные запросы (retry) с экспоненциальной задержкой при временных ошибках.
Результат:
Сервис остаётся доступным, пользователь не замечает сбоя.

### Сбой брокера сообщений (Kafka)

Проблема:
Брокер сообщений временно недоступен.
Микросервисы не могут обмениваться событиями (например, "объявление создано").

Решения (некоторые потребуют кодовой разработки):

Устойчивые к сбоям паттерны:
Сообщения хранятся на диске (Kafka: репликация; RabbitMQ: durable queues).
Репликация и кластеризация — Kafka или RabbitMQ работают в кластере из 3+ нод.
Автовосстановление — после восстановления брокера, потребители продолжают с последнего offset.

Работа продьюсеров (producers):
Локальное кэширование сообщений при недоступности брокера (если критично).
Повторные попытки отправки (retry с backoff).

Поведение консьюмеров (consumers):
Подтверждение обработки (ack) только после успешного выполнения.
При сбое — сообщение возвращается в очередь или повторяется позже.

Результат:
Данные не теряются, обработка возобновляется после восстановления.

### Сетевые сбои и задержки (network partition)

Проблема:
Сетевой разрыв между микросервисами (например, в облаке).
Запросы зависают или возвращают таймаут.

Решения (некоторые потребуют кодовой разработки):

Таймауты на всех уровнях:
HTTP-клиенты: таймаут 5–10 сек.
Брокеры сообщений: heartbeat, reconnect.
Circuit Breaker (шаблон "предохранитель"):
Пример: если 5 запросов подряд к user-service падают — цепь размыкается на 30 сек, предотвращая лавину.
Fallback-логика:
Вернуть кэшированные данные (если возможно).
Вернуть частичный ответ (например, объявление без имени автора).

Результат:
Система не "падает целиком", а деградирует по функциональности.
